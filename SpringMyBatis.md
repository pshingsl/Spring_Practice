# Spring MyBatis

## 개요


자바 객체와 SQL 사이를 자동 매핑하여 DB 작업을 간소화해주는 영속성(Persistence)프레임워크이다.

→ MyBatis는 **영속성 프레임워크라기보다는 SQL Mapper 프레임워크**이다.


ORM(Object Relational Mapping)프레임워크이다.

→ MyBatis는 **ORM이 아니라 SQL Mapper 방식**이다.


SQL을 XML이나 어노테이션으로 분리하여 직접 관리하며,
복잡한 쿼리에 유리하고 JDBC의 반복적인 코드(Connection, Statement 등)를 줄여준다.

---

## ORM을 사용하는 이유


ORM이 도입되게 전에 객체 지향 설계와 RDB간의 불일치로 인한 불편함을 겪음.

→ 객체 지향 설계와 관계형 데이터베이스(RDB) 간의 패러다임 불일치 문제를 해결하기 위해 ORM이 등장하였다.


객체 지향은 추상화, 다형성, 캡슐화 등을 통해 복잡성을 관리하지만,
RDB는 데이터 중심 구조이기 때문에 객체 개념이 부족하다.


필드를 추가해야하는 점이 있다
→ 불필요한 컬럼 또는 구조적 제약이 발생할 수 있다.


이러한 차이를 해결하기 위해 ORM이 사용된다.

---

## MyBatis 특징



* SQL을 직접 작성하고 제어할 수 있다.
* SQL과 Java 코드를 분리(XML 또는 Annotation)
* 동적 SQL 작성 가능 (조건문, 반복문 등)
* SQL 실행 흐름을 개발자가 완전히 제어

---

## 장점

### 1) SQL 중심 개발


ORM을 이용하면 SQL 쿼리가 아닌 메서드로 데이터를 조작가능
→ MyBatis는 SQL을 직접 작성한다.


복잡한 쿼리를 직접 작성하고 제어할 수 있어 성능 최적화에 유리하다.

---

### 2) 유지보수 용이성


SQL과 Java 코드가 분리되어 있어 유지보수가 쉽다.


매핑정보가 명확하여 ERD 의존도를 낮출 수 있다
→ SQL이 명확하게 드러나기 때문에 데이터 구조 이해가 쉽다.

---

### 3) 생산성


JDBC 대비 반복 코드가 줄어들어 개발 생산성이 향상된다.

---

## 단점



* SQL을 직접 작성해야 하므로 코드량 증가
* 객체와 테이블 간 불일치 해결을 직접 해야 함
* DBMS 변경 시 SQL 수정 필요 가능성 있음


ORM 설계 문제 → MyBatis는 ORM이 아니므로 해당 표현 제거

---

## 프로시저 관련


프로시저가 많은 시스템에서 ORM의 객체 지향적인 장점을 활용하기 어렵다.

→ 프로시저 기반 시스템에서는 ORM의 장점을 활용하기 어렵지만,
MyBatis는 SQL 기반이기 때문에 비교적 잘 맞는다.


프로시저는 특정 로직을 처리하는 DB 내부 프로그램이다.

---

## 참고 개념

### 영속성(Persistence)


데이터를 저장하여 프로그램이 종료되어도 유지되는 특성

---

### ORM(Object Relational Mapping)


객체와 관계형 데이터베이스 테이블을 자동으로 매핑하는 기술

---

### SQL Mapper


SQL과 객체를 매핑하는 방식으로
개발자가 SQL을 직접 작성하고 결과를 객체에 매핑한다.

---

## ORM 장단점

### 장점


* 생산성 향상 (CRUD 자동화)
* 유지보수 용이
* DB 의존성 감소

### 단점



* 학습 난이도 높음
* 복잡한 쿼리 성능 이슈 가능
* 자동 생성 SQL 최적화 어려움

---

## SQL Mapper (MyBatis 방식)

### 특징



* SQL 직접 작성
* 동적 SQL 지원
* 성능 튜닝에 유리
* DB 중심 개발 방식

---

### 장점


* 복잡한 쿼리 작성 및 최적화 용이
* SQL 제어 가능
* 기존 DB 구조 활용 용이

---

### 단점



* 객체 중심 개발 어려움
* SQL 의존성 증가
* DB 변경 시 수정 필요

---

## MyBatis 실행 흐름

### 실행 다이어그램

```
[Client 요청]
      ↓
[Controller]
      ↓
[Service]
      ↓
[Mapper Interface 호출]
      ↓
[MyBatis Framework]
      ↓
[XML / Annotation SQL 매핑 확인]
      ↓
[SQL 실행 (JDBC)]
      ↓
[DB]
      ↓
[ResultSet 반환]
      ↓
[객체 매핑]
      ↓
[Controller → JSON 응답]
```

---

### 동작 과정 설명


1. 클라이언트가 요청을 보낸다.
2. Controller에서 요청을 받아 Service로 전달한다.
3. Service는 Mapper 인터페이스를 호출한다.
4. MyBatis가 Mapper 인터페이스의 구현체를 동적으로 생성하여 실행한다.
5. XML 또는 Annotation에서 SQL을 찾아 실행한다.
6. 내부적으로 JDBC를 사용하여 DB에 SQL을 실행한다.
7. DB 결과(ResultSet)를 받아 객체로 자동 매핑한다.
8. 결과를 Controller로 반환하고 JSON 형태로 응답한다.

---

### 핵심 포인트


* MyBatis는 JDBC 기반으로 동작한다.
* SQL 실행은 JDBC에 위임된다.
* Mapper 인터페이스는 프록시 객체로 동작한다.
* 결과를 자동으로 객체로 매핑한다.

---

## 핵심 비교 (JDBC vs MyBatis vs JPA)

| 구분     | JDBC | MyBatis | JPA   |
| ------ | ---- | ------- | ----- |
| SQL 작성 | 직접   | 직접      | 자동 생성 |
| 생산성    | 낮음   | 중간      | 높음    |
| 제어     | 높음   | 매우 높음   | 낮음    |
| 객체 중심  | 낮음   | 중간      | 높음    |
| 복잡한 쿼리 | 어려움  | 강함      | 약함    |

---

## 핵심 한줄 정리


MyBatis는 "SQL을 직접 제어하면서 객체와 매핑하는 SQL Mapper 기술"이다.

---

## 학습 흐름 연결


JDBC → MyBatis → JPA 순으로 학습하면
데이터 접근 방식의 발전 과정을 이해할 수 있다.
